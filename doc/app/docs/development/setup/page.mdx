# Getting Started

This guide covers the initial setup of your development environment and explains the core project structure of RedisHub.

## Prerequisites

Ensure you have the following tools installed and available in your `PATH`:

| Tool | Version | Notes |
|------|---------|-------|
| **Go** | 1.26+ | Core backend logic. |
| **Node.js** | 22+ | Frontend runtime. |
| **pnpm** | 10.3+ | Fast package manager (corepack enable pnpm). |
| **Docker** | recent | Required for local Redis instances. |
| **scorix CLI** | latest | `go install github.com/tradalab/scorix/cmd/scorix@latest` — drives `make dev/generate/build/package`. |

### Runtime Dependencies

These are loaded by the app at startup — they are **not** build dependencies:

- **Windows** — [WebView2 Evergreen Runtime](https://developer.microsoft.com/microsoft-edge/webview2/) (preinstalled on Windows 11 / recent Windows 10).
- **Linux** — GTK3 + WebKitGTK: `sudo apt-get install -y libgtk-3-0 libwebkit2gtk-4.1-0` (falls back to `4.0`).
- **macOS** — none; WKWebView ships with the OS.

> [!TIP]
> Run `make doctor` (`scorix doctor`) to verify your toolchain and runtime dependencies.

## Workspace Initialization

1.  **Clone the repository**:
    ```bash
    git clone https://github.com/tradalab/redishub.git
    cd redishub
    ```
2.  **Install dependencies**:
    ```bash
    make deps
    ```

## Local Redis Instances

RedisHub provides a pre-configured Docker Compose setup for testing different topologies:

```bash
make redis-up
```

| Service | Port | Description |
|---------|------|-------------|
| **Standalone** | `6001` | Basic single instance. |
| **Sentinel** | `9001` | High-availability discovery node. |
| **Cluster** | `7001-7003` | Sharded Redis nodes. |

> [!NOTE]
> For the cluster setup, run `make redis-init-cluster` once after the containers are up to initialize the sharding.

## Running Development Server

To start the Go backend and the Next.js dev server (with HMR) together:

```bash
make dev
```

This runs `scorix dev`: the application window opens automatically, frontend edits hot-reload live,
and IPC calls to the Go backend work end-to-end.

## Project Structure

| Path | Description |
| :--- | :--- |
| `main.go` | Entry point — wires the app + browser/updater modules; selects `app` or `web` mode. |
| `proto/app.proto` | IPC contract. Handlers (`internal/handler`) and types (`internal/types`) are **generated** from it. |
| `etc/schema.sql` | DB schema. sqlx CRUD models (`internal/model`) are **generated** from it. |
| `scorix.yaml` | Single config source: build recipe + runtime manifest (app, window, logger, modules). |
| `internal/` | Backend: generated `handler`/`types`, `logic/<service>`, `model` (sqlx), `svc`, `config`. |
| `pkg/` | Reusable Go packages (`netx` SSH/proxy dialers, `util` key encoding). |
| `shell/` | The Next.js frontend (React, Tailwind, Shadcn UI), built into `.scorix/dist`. |
| `doc/` | The Nextra-powered documentation site (what you are reading now). |
| `scripts/` | `release.sh` — version bump helper. |
| `dev/` | Docker Compose files for local Redis topologies. |
| `docker/` | `Dockerfile_web` — the web/server image (built & pushed by CI). |
