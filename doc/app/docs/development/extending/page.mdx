# Adding a New IPC Command

RedisHub bridges the Go backend and the Next.js frontend with an **Inter-Process Communication (IPC)**
layer that is **generated from a Protobuf contract**. Commands follow the pattern `service:method`
(e.g. `client:connect`, emitted from the proto `rpc` `client.Connect`).

You do **not** hand-write the command router. `internal/handler/handler.go` and
`internal/types/types.go` are generated — each carries a `// Code generated by scorix. DO NOT EDIT.`
header. To add a command you edit the proto, regenerate, then implement the logic stub.

---

## Step 1: Declare the Contract in `proto/app.proto`

Add the request/response messages and an `rpc` to the relevant `service`. Use `returns (Empty)` for a
fire-and-forget call, a response message for a request/response call, or `returns (stream …)` for a
server-streaming command (like `monitor:Start` or `console:Exec`).

```proto
message MyActionReq {
  string connection_id  = 1;
  int32  database_index = 2;
  string custom_param   = 3;
}

message MyActionRes {
  string result = 1;
}

service client {
  // ... existing rpcs
  rpc MyAction(MyActionReq) returns (MyActionRes);
}
```

---

## Step 2: Regenerate Handlers, Types & Models

```bash
make generate
```

This runs `scorix generate proto` and `scorix generate model`. From the proto it regenerates the Go
handler (`internal/handler/handler.go`), the Go types (`internal/types`), **and** the typed frontend
client (`shell/api/index.ts`). From `etc/schema.sql` it regenerates the sqlx models
(`internal/model`). The new command is now registered on both sides.

---

## Step 3: Implement the Business Logic

Create `internal/logic/<service>/<action>_logic.go`. The generated handler constructs your logic with
`(ctx, svcCtx)` and calls the method with the decoded request:

```go
package client

import (
	"context"

	"github.com/tradalab/rdms/internal/svc"
	"github.com/tradalab/rdms/internal/types"
)

type MyActionLogic struct {
	ctx    context.Context
	svcCtx *svc.ServiceContext
}

func NewMyActionLogic(ctx context.Context, svcCtx *svc.ServiceContext) *MyActionLogic {
	return &MyActionLogic{ctx: ctx, svcCtx: svcCtx}
}

func (l *MyActionLogic) MyAction(req *types.MyActionReq) (any, error) {
	// 1. Resolve the Redis client via l.svcCtx.RedisManager
	// 2. Perform the operation
	// 3. Return a serializable result (or *types.MyActionRes) and error
	return &types.MyActionRes{Result: "ok"}, nil
}
```

> [!NOTE]
> For a **server-streaming** rpc, the generated handler wires an `app.Sink[T]`; implement
> `MyAction(req *types.MyActionReq, out app.Sink[types.MyEvent]) error` and push frames with
> `out.Send(...)`. See `internal/logic/monitor/start_logic.go` for a worked example.

---

## Step 4: Call from the Frontend

Use the generated, typed client in `shell/api/index.ts`. The proto `rpc` `client.MyAction` is exposed
as `client.myAction(params)` (wire name `client:my-action`):

```typescript
import { client } from "@/api";

const res = await client.myAction({
  connection_id: "123",
  database_index: 0,
  custom_param: "hello",
});
```

Streaming commands return an async-iterable instead — e.g. `monitor.start(params)` yields frames
(`for await (const frame of monitor.start(...))`). Under the hood everything goes through the
auto-injected `window.scorix` bridge (`scorix.invoke` / `scorix.serverStream`).

> [!TIP]
> Keep the proto the single source of truth — changing a field means editing `proto/app.proto` and
> re-running `make generate`, never hand-editing the generated `handler.go` / `types.go` /
> `shell/api/index.ts`.
